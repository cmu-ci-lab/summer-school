ICCP Summer School OCT Guide

How to use the OCT rig: install the environment, verify the stage and camera, align with the live visualizer, then capture a Z-stack and turn it into a depth map.

setup.sh→ test_hardware.py→ visualizer.py→ oct_scan.py→ oct_process.py

§How the measurement works

OCT — optical coherence tomography — measures surface height with light. The rig is a low-coherence interferometer: light from a broadband source is split between a reference path and the sample, then recombined on the camera. Because the source's coherence length is short, interference fringes appear at a pixel only when the two paths match to within a few tens of microns. Everywhere else, the pixel just sees the ordinary incoherent image.

The motorized stage sweeps that plane of zero path difference — the coherence gate — through the sample. As it passes a pixel's surface, that pixel briefly flickers with fringes; before and after, it's quiet. So the measurement is simply: record a frame at every stage position (a Z-stack), then find, per pixel, the position where the fringe amplitude peaked. That position is the pixel's height — a full depth map from one sweep. The amplitude-vs-position curve for a pixel is the coherence function: its peak marks the depth, and its width (the coherence FWHM) sets the axial resolution — you can't localize a surface more finely than the gate is thick.

Everything in this guide serves that one measurement: the visualizer plots the coherence function for a patch live so you can find the gate and judge signal quality, oct_scan.py records the sweep, and oct_process.py finds the per-pixel peak.

1Installation

setup.sh · environment.yml · requirements.txt

The rig is a Thorlabs KDC101-driven motorized stage plus a monochrome machine-vision camera — either Allied Vision (via the Vimba X SDK) or IDS uEye (via pyueye). All the software runs from one Python environment, and one script builds it on Windows (Git Bash) and macOS.

Getting the code

Every script in this guide lives in the course repository at github.com/cmu-ci-lab/summer-school. Clone it and run everything from inside it:

git clone https://github.com/cmu-ci-lab/summer-school.git
cd summer-school

Running setup.sh

The script prefers conda if you have it and falls back to a plain .venv if you don't:

chmod +x setup.sh
./setup.sh
# or, to force a venv even when conda exists:
FORCE_VENV=1 ./setup.sh

It creates or updates the iccp-oct environment from environment.yml (Python 3.12, numpy, scipy, matplotlib, OpenCV, pyserial, pylablib, pyftdi), registers a Jupyter kernel, and prints activation instructions when it finishes:

# conda path
conda activate iccp-oct
# venv path
source .venv/bin/activate        # Windows: .venv\Scripts\activate

In VSCode, pick the interpreter with Ctrl/Cmd+Shift+P → Python: Select Interpreter → Python (iccp-oct) (or ./.venv). Some scripts can also be run cell-by-cell in the Interactive Window, so it's worth setting this up now.

After the environment, install the SDK for your camera — one of the next two subsections applies, not both.

Allied Vision cameras: the vmbpy wheel

vmbpy is not on PyPI — it ships as a wheel inside the Vimba X SDK. setup.sh searches the standard install locations and installs the wheel automatically if it finds one; otherwise install Vimba X first, then re-run setup.sh (or pip install the wheel directly):

Install the wheel into the project environment (activate iccp-oct or the venv first, then pip install <path-to-vmbpy-*.whl>) — not into the system Python, or imports will fail when you run from the project env.

IDS cameras: pyueye

IDS uEye cameras need two pieces:

  1. The IDS Software Suite (the uEye SDK and driver) from ids-imaging.com — this is the native driver the Python bindings talk to. Without it, pyueye imports but every call fails.
  2. The Python bindings: pip install pyueye into the project environment.

That's it — no wheel hunting. If you're on IDS hardware you can skip Vimba X entirely; pass --camera ids to the hardware test so it doesn't check for vmbpy.

Platform notes (already handled by setup.sh)

  • macOS: Apple's FTDI driver ignores the KDC101's custom PID (0xfaf0), so no /dev/cu.usbserial-* port ever appears. Don't hunt for one — the stage is driven directly over libusb via pyftdi. setup.sh installs libusb (conda-forge or Homebrew) for you.
  • Windows: the Thorlabs Kinesis/APT driver must be installed for the stage.

2Sanity checking

test_hardware.py

Before doing anything scientific, prove the machine can talk to both devices.

Running the test

python test_hardware.py                 # auto-detect camera (AVT, then IDS)
python test_hardware.py --camera ids    # IDS only — skips the Vimba X check
python test_hardware.py --camera avt    # Allied Vision only

The stage test connects, reads the position, moves +5 mm and back, and verifies the readback — so make sure the stage has 5 mm of clearance before you run it. The camera test connects, grabs a frame, buffers three more, and writes a small stack to test_captures/.

What passing looks like

Each step prints a checkmark as it succeeds; a healthy run looks like this:

  SUMMARY
  Stage:  ✓ PASS
  Camera: ✓ PASS

Things to check in that output beyond the checkmarks:

  • Position error should be well under 0.1 mm — a ⚠ position-mismatch warning means the stage moved but the readback disagrees (loose coupling, wrong stage model, or motion not finished).
  • Frame shape and dtype should match your sensor — full resolution, and uint16 if the camera runs a 10/12-bit mode (IDS) rather than 8-bit.
  • The saved stack size should be plausible (a few frames × resolution × 1–2 bytes).

The exit code is a bitfield — 0 all good, 1 stage failed, 2 camera failed, 3 both — handy if you're scripting.

Stage not found? Check the USB cable and that the KDC101 is powered on. On macOS remember: no serial port is expected (see §1), and libusb must be installed. If a script crashed earlier, the hardware may still be held by its Python kernel — restart the kernel and retry.

3Hardware operation

stage.py · camera.py

Every script in the repo drives the rig through two small classes: ThorlabsStage and Camera. You never talk to the vendor SDKs directly — and if you want to write your own scan or alignment script, this is the whole API you need.

The stage: ThorlabsStage

from stage import ThorlabsStage

stage = ThorlabsStage(units="mm")
stage.connect()
stage.home()               # find the hardware zero — once per session
stage.move_to(1.50)        # absolute move; blocks until it arrives
stage.move_by(0.001)       # relative move
print(stage.position)      # true readback position, in mm
stage.release()
MemberWhat it does
connect() / release()Open / close the KDC101 connection
home(wait=True)Drive to the reference switch so positions are absolute. Do it once per session — oct_scan.py does it for you
move_to(pos, wait=True)Absolute move in mm; blocks until done unless wait=False
move_by(dist, wait=True)Relative move (negative = toward home)
positionEncoder readback in mm — the true position. The scan script records this per frame rather than trusting the target
is_moving / wait_move() / stop()Poll, block on, or abort a motion in progress

Pass wait=False to move while doing something else — that's how the visualizer keeps the preview running during a sweep, polling position and is_moving as it goes.

The camera: Camera

from camera import Camera

cam = Camera(exposure_us=8000, save_dir="my_scan")   # auto-detects AVT, then IDS
cam.connect()
frame = cam.capture()         # one (H, W) numpy frame, uint8 or uint16
cam.capture_to_buffer()       # accumulate frames in memory...
cam.save_stack("stack.npy")   # ...then write them as one (H, W, N) array
cam.release()

Camera is a thin auto-detecting wrapper: it tries Allied Vision first, then IDS, and forwards every method to whichever backend connected (avt_camera.py or ids_camera.py). Pass prefer="ids" or prefer="avt" to skip detection, serial= to pick a specific IDS camera, or ip= for a GigE Allied Vision camera.

Allied Vision cameras additionally support set_binning() and continuous streaming (start_streaming() / latest_frame()) — the visualizer uses them when available and falls back to plain capture() when not.

Connecting & releasing

Both classes are context managers, so for your own scripts the safest pattern is:

with ThorlabsStage(units="mm") as stage, Camera(exposure_us=8000) as cam:
    stage.move_to(1.5)
    cam.capture()
# both released automatically, even if something throws

"Device busy" or a camera/stage that won't connect? Only one process can hold each device. A script that crashed without release() keeps the handle — restart its Python kernel (or close the other program), then reconnect. On IDS, a failed is_InitCamera usually means exactly this, or that the IDS Software Suite isn't installed.

4The visualizer

visualizer.py

The visualizer is the alignment tool: a streaming camera preview with keyboard stage control, plus a side panel where you pick a patch of pixels and its interference amplitude is plotted against stage position in real time. Use it to get your first image on screen, set the exposure, and locate the coherence function before committing to a scan. If no stage is connected it falls back to a camera-only preview.

python visualizer.py
python visualizer.py --step 0.1 --window-mm 0.2
python visualizer.py --binning 4 --exposure 10000

Anatomy of the window

Live viewing basics

Three concepts behind the preview, worth understanding because they show up everywhere:

  • Binning. The preview uses 4×4 on-sensor binning by default (--binning) for a fast frame rate. Binning sums pixels, so a binned image is ~N² brighter. Binning is reset to 1 when you quit, so full-resolution captures aren't affected.
  • Non-binned-equivalent exposure. Because of that brightness gain, the tool divides the real exposure by binning² and displays the equivalent full-resolution value. The number you dial in with +/− is therefore directly reusable as --exposure in oct_scan.py, which captures at full resolution.
  • Display gamma. The sensor is linear; your eye is not. The preview applies gamma 2.2 by default so dark scenes are visible (--gamma 1.0 for a linear view). This affects the display only, never the data.

First session goal: frame the sample, find an exposure where nothing saturates, and note the rough stage region where fringes appear — a fine banding or flicker in the image that shows up only while the surface sits in the coherence gate, and vanishes again as you move past it. Write the exposure down — you'll reuse it in oct_scan.py.

The coherence function

For your selected patch, each new frame is scored as the mean absolute deviation from the local DC mean — the same compute_mean_diff measure the OCT processing pipeline uses (oct.py). "Local" is positional, not temporal: the DC mean comes from patches captured within ±--window-mm (default 0.2 mm) of the current stage position. The intuition: the positional mean is the ordinary incoherent image — what the patch looks like with no interference — and the fringes oscillate around it as the stage moves, so the mean |frame − mean| over the patch is an estimate of the fringe amplitude. That makes the plot robust to irregular back-and-forth motion, and it keeps exactly one value per position, so you can sweep over the same region repeatedly and refine the curve rather than smear it.

Once the curve has crossed half-max on both sides of the peak, the panel reports two numbers in the stat tiles: the peak position (the midpoint of the two half-max crossings — the depth of your patch) and the coherence FWHM (the width between the crossings, after subtracting the background floor — the axial resolution of the system).

Reading the coherence function — a practical recipe

  1. Pick a patch on structure. Select a small region with actual texture or fringes. A patch on a featureless background has nothing to modulate and the curve will stay flat.
  2. Watch for saturation. If pixels clip at the top of the range, the amplitude measure is distorted. Lower the exposure until the brightest fringes stay below clipping.
  3. Sweep coarse, then fine. Hold w/s with a large step to cross the whole travel range and spot where the curve lifts off the floor; then shrink the step with [ and re-sweep the bump slowly with e/d so the coherence function gets dense sampling.
  4. Wait for the readout. The coherence FWHM and peak tiles fill in once the curve has crossed half-max on both sides of the peak — you must sweep through the whole coherence function, not just up one flank.
  5. Note the peak position. That's the center of the region your scan should straddle. If the coherence function looks noisy, press r and re-sweep more slowly rather than trusting a ragged curve.

Tuning --window-mm: it sets the positional width used for the DC (local mean) estimate. If it's much wider than the coherence function, the DC estimate is fine; if you're working with a very narrow coherence function, shrink it so the local mean isn't contaminated by the fringes themselves.

5Scanning & processing

oct_scan.py · read_stack.py · oct_process.py · oct_view.py

Once you know the exposure and where the coherence function peaks, record the sweep as a Z-stack and process it into an OCT depth map.

Capturing a stack: oct_scan.py

oct_scan.py homes the stage, sweeps it in fixed steps, captures one frame per position, and saves everything. You give it the three numbers that define the scan — start position, step size, and frame count:

python oct_scan.py --start 1.50 --step-mm 0.001 --frames 400
python oct_scan.py --start 1.50 --step-mm 0.002 --frames 200 \
    --exposure 8000 --save-dir coin_scan
python oct_scan.py ... --no-home     # stage already homed this session
OptionMeaningDefault
--startStart position in mm — set it so the sweep straddles the coherence-function peak from the visualizer, with margin on both sides1.50
--step-mmStage step per frame in mm. Sample the coherence function finely: several frames per coherence FWHM0.001
-n, --framesNumber of frames — total range = frames × step400
--exposureExposure in µs — reuse the value from the visualizer8000
--save-dirOutput directory — use a fresh name per scanoct_scans
--cameraauto, avt, or idsauto
--no-homeSkip homing (faster when re-scanning in one session)—

It prints the scan plan up front (400 frames × 0.001 mm (1.5 → 1.899 mm)) and the readback position per frame, then writes two files into --save-dir:

  • a timestamped stack_*.npy — the frames as one (H, W, N) array;
  • a stack_*_meta.json sidecar — step size, start position, the true per-frame positions, exposure, and pixel pitch, so the processing scripts know the scan geometry without touching the hardware.

The stage and camera are released automatically when the scan finishes (or fails). Example: the defaults sweep 400 frames × 1 µm from 1.50 mm — a 0.4 mm range.

Checking the stack before you leave the rig

Turn it into a movie and scrub through it — you should see fringes wash across the scene:

python read_stack.py -s oct_scans/stack_20260708_1.npy --fps 30
python read_stack.py --lossy -o preview.mp4     # smaller, compressed

Computing depth: oct_process.py

oct_process.py loads a stack, computes the per-pixel interference amplitude along the sweep with compute_mean_diff (from oct.py), and takes the argmax along the frame axis as depth:

python oct_process.py -s oct_scans/stack_20260708_1.npy
python oct_process.py -s ...stack.npy -n 4     # 4x4 spatial downsampling first

It writes two arrays next to the stack, then renders and saves a depth-map PNG:

  • *_depth.npy — per-pixel depth as a frame index — convert to mm by multiplying by the step size × the frame stride (2; see the every-second-frame note below);
  • *_maxamp.npy — the coherence function's peak amplitude per pixel, a confidence map: where it's near the noise floor, the depth value is meaningless.

Practical knobs:

  • -n / --downsample averages N×N pixel patches before the computation. It's the first thing to reach for if the full-resolution run is slow or runs out of memory — and the averaging suppresses noise, too. Outputs are tagged (_ds4_depth.npy) so runs don't overwrite each other.
  • The script feeds every second frame (frames[:, :, ::2]) into the computation to halve the work — so one depth index equals two stage steps when you convert to mm.
  • It's also cell-structured (#%%), so in the Interactive Window you can recompute the plot without reloading the stack.

Re-rendering depth maps

The depth computation is the slow part; the colormap is not. oct_view.py loads an existing *_depth.npy and just re-renders it. Edit the constants at the top:

DEPTH_PATH = Path("oct_scans/stack_..._depth.npy")
DEPTH_MIN  = 60      # clip the colormap — or None for auto
DEPTH_MAX  = 80
CMAP       = "viridis"

Always tune DEPTH_MIN/DEPTH_MAX. Pixels with no signal argmax to arbitrary frames and stretch the auto colormap until the real surface is one flat color. Clip the range to the frames where your object actually lives (check the printed data range, or mask by *_maxamp.npy) and the relief pops out.
