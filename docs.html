Environment Setup

All code runs inside a project-local conda environment named iccp. Three files manage this:

environment.yml

Conda env spec — Python 3.12 + all pip dependencies

requirements.txt

Pip-only list for reference or non-conda installs

setup.sh

One-shot setup script (Git Bash on Windows)

One-shot setup (Git Bash)

./setup.sh

The script will: create the iccp conda environment, install all pip packages, install vmbpy from the local Vimba X SDK wheel, and register the Jupyter kernel for VSCode.

Manual setup

# 1. Create environment
conda env create -f environment.yml

# 2. Install vmbpy from Vimba X SDK wheel
conda activate iccp
pip install "C:/Program Files/Allied Vision/Vimba X/api/python/vmbpy-1.1.0-py3-none-win_amd64.whl"

# 3. Register Jupyter kernel (for VSCode interactive window)
python -m ipykernel install --user --name iccp --display-name "Python (iccp)"

ThorlabsStage stage.py

Wraps pylablib.devices.Thorlabs.KinesisMotor. Auto-detects the first connected Brushed Motor Controller and reads the stage type from device EEPROM (scale="stage") — so the correct encoder scale is applied automatically regardless of which compatible stage is attached.

Basic usage

from stage import ThorlabsStage

with ThorlabsStage(units="mm") as stage:
    stage.home()
    stage.move_to(10)           # absolute, 10 mm
    stage.move_by(0.5)          # relative, +0.5 mm
    print(stage.position)       # current position in mm

Unit selection

Pass units="m" or units="mm" at construction. All inputs and outputs (move_to, move_by, position) use the chosen unit. Internally pylablib always works in metres; the class multiplies/divides transparently.

AVTCamera capture_camera.py

Wraps vmbpy. On connect it scans all cameras, filters for Allied Vision by vendor name, opens the first match, and creates save_dir if it doesn't exist.

Single frame

from capture_camera import AVTCamera

with AVTCamera(exposure_us=10000, gain_db=0.0, save_dir="captures") as cam:
    image = cam.capture()          # raw uint8 or uint16, shape (H, W)
    image8 = cam.capture_8bit()    # rescaled uint8
    cam.save(image, cam.timestamped_filename())

Buffered stack acquisition

Use this pattern when stepping the stage between frames. Each call to capture_to_buffer() appends to an internal list. save_stack() stacks them to (H, W, N), saves as .npy, and clears the buffer.

for i in range(200):
    stage.move_to(i * 0.005 + 18)    # 5 µm steps from 18 mm
    cam.capture_to_buffer()

cam.save_stack(
    cam.timestamped_filename(prefix="stack", ext="npy"),
    save_previews=True,             # also writes 8-bit PNGs to captures/captured_images/
)

Saving behaviour

Input dtypeExtensionOutput
any.npyRaw numpy array saved as-is
uint16.png / .tiff16-bit TIFF (full precision) + rescaled 8-bit PNG
uint8.pngStandard 8-bit PNG
💡
Call cam.clear_buffer() to discard frames without saving — useful if a run is aborted midway.

Depth-from-Focus depth_from_focus.py

Translated from MATLAB (computeMeanDiff.m + getintAmplitude.m). Measures local intensity modulation per frame — the frame with highest modulation at each pixel is the in-focus frame, giving the depth map.

Algorithm

  1. DC estimate — either global mean across all frames, or a 1-D temporal box filter of width temporal_window (running average). Boundary frames are replicated rather than zero-padded to match the original MATLAB behaviour.
  2. AC component — |frame − DC| per pixel per frame.
  3. Spatial smoothing — 2-D box filter of size patch_size × patch_size applied independently to each frame slice.
  4. Depth — argmax along the frame axis → index of best-focus frame per pixel.
from depth_from_focus import compute_mean_diff
import numpy as np

frames = np.load("captures/stack.npy")       # (H, W, N)

_, md_image, _ = compute_mean_diff(
    frames[:, :, ::2],          # every other frame
    patch_size=5,
    avg_type="local",
    temporal_window=20,
)

depth  = np.argmax(md_image, axis=2)   # (H, W) — best-focus frame index
maxamp = np.max(md_image,   axis=2)   # (H, W) — modulation amplitude
ℹ️
depth is 0-indexed in Python (MATLAB returns 1-indexed). For visualisation this makes no difference; if you need physical depth in mm multiply by your step size: depth_mm = depth * 0.005.

Acquisition Script move_stage.py

Connects both devices, homes the stage, sweeps in 5 µm steps, and saves a timestamped .npy stack with 8-bit preview PNGs.

from stage import ThorlabsStage
from capture_camera import AVTCamera

stage = ThorlabsStage(units="mm")
stage.connect()

cam = AVTCamera(exposure_us=10000, gain_db=0.0, save_dir="coin_captures")
cam.connect()

stage.home()

for i in range(200):
    stage.move_to(i * 0.005 + 18)
    cam.capture_to_buffer()
    print(f"  {stage.position:.3f} mm")

cam.save_stack(cam.timestamped_filename(prefix="stack", ext="npy"), save_previews=True)

stage.release()
cam.release()

Outputs written to coin_captures/:

  • stack_YYYYMMDD_HHMMSS.npy — raw frame stack, shape (H, W, 200)
  • captured_images/stack_*_frame0000.png … — rescaled 8-bit previews

Processing Script run_depth.py

Loads a saved stack and runs the depth-from-focus pipeline.

from depth_from_focus import compute_mean_diff
import numpy as np
import matplotlib.pyplot as plt
from pathlib import Path

stack_path = Path("coin_captures/stack.npy")
frames = np.load(stack_path)

_, md_image, _ = compute_mean_diff(
    frames[:, :, ::2], patch_size=5, avg_type="local", temporal_window=20
)

depth  = np.argmax(md_image, axis=2)
maxamp = np.max(md_image,   axis=2)

np.save(stack_path.with_name(stack_path.stem + "_depth.npy"),  depth)
np.save(stack_path.with_name(stack_path.stem + "_maxamp.npy"), maxamp)

Outputs (saved alongside the input stack):

  • stack_depth.npy / stack_depth.png — depth map (viridis colourmap)
  • stack_maxamp.npy / stack_maxamp.png — max amplitude (inferno colourmap)

Docker — useful?

Short answer: no for acquisition, possibly yes for processing.

This system controls physical USB hardware using proprietary Windows drivers (Thorlabs APT and Allied Vision Vimba X). Docker on Windows cannot realistically support this:

BlockerWhy it matters
Thorlabs APT USB driverWindows kernel driver — cannot run inside a Linux container. Even with USB passthrough via usbipd-win, the APT protocol driver won't be present in the container.
Vimba X SDKShips as a Windows installer; the vmbpy wheel and underlying transport layer DLLs are Windows-only.
USB passthrough complexityDocker Desktop on Windows routes USB through WSL2 via usbipd-win — brittle, requires manual attachment each session, breaks on reconnect.
⚠️
Do not containerise the acquisition side (move_stage.py). The conda environment + setup.sh already solves reproducibility for the hardware layer on Windows.

Where Docker does make sense

The processing pipeline (depth_from_focus.py, run_depth.py) is pure Python + numpy/scipy/matplotlib with no hardware dependency. A Docker image here buys you:

  • Running depth reconstruction on a Linux server or HPC cluster from a saved .npy stack
  • Sharing results with collaborators who don't have the hardware
  • Pinning exact library versions for reproducible paper results

A minimal processing-only Dockerfile would look like:

FROM python:3.12-slim
WORKDIR /app
COPY requirements_processing.txt .
RUN pip install --no-cache-dir -r requirements_processing.txt
COPY depth_from_focus.py run_depth.py ./
# mount your data at runtime:
# docker run -v /path/to/captures:/app/captures my-iccp-image python run_depth.py

Where requirements_processing.txt contains only numpy, scipy, and matplotlib — no hardware SDKs needed.

API Reference — ThorlabsStage

Method / PropertyDescription
ThorlabsStage(units="mm")Constructor. units: "mm" or "m".
.connect()Find and open the first Brushed Motor Controller. Returns self.
.release()Close the device connection.
.home(wait=True)Home the stage; blocks until complete if wait=True.
.move_to(pos, wait=True)Absolute move in user units.
.move_by(dist, wait=True)Relative move in user units.
.stop()Immediately stop motion.
.positionRead current position in user units (property).
.is_movingTrue while motion is in progress (property).
.wait_move()Block until current move completes.

API Reference — depth_from_focus

FunctionDescription
compute_mean_diff(frames, patch_size, avg_type, temporal_window)Main entry point. Returns (md_vector, md_image, blurred). md_image is (H, W, N) — take argmax over axis 2 for depth.

💡
To convert the depth index to physical depth in mm: depth_mm = depth * step_size_mm, where step_size_mm = 0.005 for the 5 µm steps used in the acquisition script.
